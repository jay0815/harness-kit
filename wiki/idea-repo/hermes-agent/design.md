# Hermes Agent 设计决策

本文档记录 Hermes Agent 的 10 项关键设计决策及其 rationale。

---

## 1. 可插拔上下文引擎（ABC 模式）

### 决策

定义 `ContextEngine` 抽象基类，包含 5 个生命周期方法：

| 方法 | 触发时机 |
|---|---|
| `onSessionStart` | 会话开始时 |
| `updateFromResponse` | 每轮收到模型响应后 |
| `shouldCompress` | 发送请求前检查上下文长度 |
| `compress` | `shouldCompress` 返回 True 时 |
| `onSessionEnd` | 会话结束时 |

默认实现为 `ContextCompressor`，参数：

| 参数 | 默认值 | 含义 |
|---|---|---|
| `threshold_percent` | 0.75 | 上下文超过 max_tokens 的 75% 时触发压缩 |
| `protect_first_n` | 3 | 始终保留前 N 条消息（含系统提示） |
| `protect_last_n` | 6 | 始终保留最近 N 条消息 |

### 为什么用 ABC

压缩策略与场景强相关：长文档分析需要摘要化早期内容，代码生成需要保留结构化代码块，多轮对话需要保留关键决策点。ABC 模式允许在不改动 Agent 循环的前提下替换策略。

### 插件目录

自定义引擎放在 `plugins/context_engine/`，通过配置名自动加载。

## 2. 每轮工具护栏（vs 简单循环检测）

### 决策

`ToolCallGuardrailController` 在**每轮对话**级别跟踪工具调用，而非全局计数。

### 检测模式

| 模式 | 检测内容 | 动作 |
|---|---|---|
| 相同失败重复 | 同一工具 + 同一参数 + 同一错误 | `halt`（终止本轮） |
| 同工具失败链 | 同一工具连续失败（参数可变） | `block`（阻止该工具） |
| 无进展幂等调用 | 幂等工具返回与之前相同结果 | `warn`（警告） |

### 工具分类

| 分类 | 特征 | 示例 |
|---|---|---|
| **IDEMPOTENT** | 多次调用结果相同，无副作用 | `read_file`, `search_files`, `web_search` |
| **MUTATING** | 会改变系统状态 | `terminal`, `write_file`, `patch` |

### 动作层级

`allow` → `warn` → `block` → `halt`

### 为什么用每轮而非全局

全局计数器（如"最多 50 次工具调用"）无法区分"有意义的重复"和"无意义的循环"。每轮护栏在更细粒度上检测：幂等工具的"无进展"调用、变更工具的失败链扩散，在问题发生前即介入。

## 3. 自注册工具发现（AST 解析）

### 决策

`ToolRegistry` 单例通过 **AST 静态解析**发现模块顶层的 `registry.register()` 调用，无需显式注册列表。

### 流程

```
读取模块源码 → 解析为 AST → 遍历 Call 节点 → 匹配 registry.register → 提取参数 → 注册
```

### 关键机制

- **generation 计数器**：每次注册/卸载时递增，用于缓存失效
- **TTL 缓存 check_fn**：30 秒缓存，避免重复验证工具签名

### 为什么用 AST

| 方式 | 优点 | 缺点 |
|---|---|---|
| 运行时反射 | 简单 | 必须导入模块，有副作用和循环导入风险 |
| AST 静态解析 | 不执行代码即可发现，无副作用 | 实现复杂，需处理语法边界 |

AST 方式避免了运行时导入所有模块的开销和副作用，支持跨模块安全扫描。

## 4. 记忆提供者单例限制

### 决策

`MemoryManager` 强制限制：**最多同时激活 1 个外部记忆提供者**。

### 生命周期

```
initialize(config)      # 会话开始前一次性初始化
    ↓
prefetch(session_id)    # 每会话开始时预取相关记忆
    ↓
sync_turn(turn_data)    # 每轮结束后同步记忆
    ↓
get_tool_schemas()      # 按需返回记忆操作工具（模型可主动查询记忆）
```

### 可选钩子

| 钩子 | 触发时机 |
|---|---|
| `on_turn_start` | 每轮开始时 |
| `on_session_end` | 会话结束时 |
| `on_pre_compress` | 上下文压缩前 |
| `on_delegation` | 子代理委派时 |

### 为什么限制为 1 个

每个外部记忆提供者都会向模型暴露自己的工具（如 `honcho_search_memory`、`mem0_add_memory`）。多个提供者并存会导致工具 schema 膨胀，增加模型调用成本，且可能产生冲突的记忆写入。内置记忆（SQLite 本地存储）不受此限制。

## 5. 提示缓存策略

### 决策

对 Claude 模型自动注入 Anthropic `cache_control` 实现**前缀缓存**。

### 缓存标记

| 内容 | cache_control | 原因 |
|---|---|---|
| 系统提示 | `ephemeral` | 会话级不变，但会话间不同 |
| 早期消息（前缀） | `persistent` | 多轮对话中不变，最大化缓存命中 |
| 最近 2-3 轮消息 | 无标记 | 每轮新增，不参与缓存 |

### 成本影响

前缀部分命中缓存后按缓存读取费率计费（约 10%），实测降低约 **75%** 的输入 token 成本。

### 自动检测

在原生 Anthropic API 和 OpenRouter 上自动检测 Claude 模型并启用缓存。系统提示的缓存状态按会话持久化到 SQLite，确保网关模式下会话重建时缓存连续性。

## 6. 错误恢复层次结构

### 决策

按优先级建立**分层恢复链**，每种错误类型有独立的恢复路径。

| 优先级 | 错误类型 | 恢复策略 |
|---|---|---|
| 1 | `UnicodeEncodeError` | 两轮清理：第一轮移除代理对（surrogates），第二轮替换不可编码字符 |
| 2 | 图片被拒绝 | 从消息中剥离所有图片附件，标记 vision-unsupported |
| 3 | 413 Payload Too Large | 紧急压缩上下文到安全大小，重试（最多 3 次） |
| 4 | 上下文溢出 | 逐步下调 context length，然后触发正常压缩 |
| 5 | 429 速率限制 | 尊重 Retry-After 头，启用 eager fallback（切换备用提供者） |
| 6 | 401/403 认证失败 | 提供者特定的凭证刷新，失败后中止 |

### 为什么分层

不同错误需要不同修复：编码错误需要内容清理，413 需要截断，限流需要等待。统一重试策略会浪费请求配额或加剧限流。分层确保每个错误走最适合的恢复路径。

## 7. 空响应恢复

### 决策

建立**渐进式空响应恢复链**，按策略优先级依次尝试。

| 策略 | 动作 | 适用场景 |
|---|---|---|
| Post-tool 轻推 | 注入合成用户消息（"请继续"） | 工具执行后模型未继续生成 |
| Thinking-only 预填充 | 预设 assistant 消息开头（如"<thinking>"） | 推理模型（如 Claude 3.7 Sonnet）的思考中断 |
| 回退提供者 | 切换到备用 API | 主提供者临时故障 |
| 空哨兵 | 返回 `_empty_terminal_sentinel` 标记 | 所有策略均失败，向调用方报告 |

### 为什么多种策略

空响应的原因各不相同：可能是模型中途停止、推理流中断、API 临时故障、或上下文导致模型无输出。单一策略无法覆盖所有情况。轻推解决"忘记继续"，预填充解决"思考中断"，回退解决"服务故障"。

## 8. 流式优先设计

### 决策

**始终优先使用流式 API**，即使没有消费者也走流式路径。

### 超时参数

| 参数 | 值 | 说明 |
|---|---|---|
| stale-stream 检测 | 90 秒 | 超过 90 秒无数据视为 stale stream |
| read 超时 | 60 秒 | 单次读取超时 |

### 为什么流式优先

即使调用方不需要实时输出，流式路径提供了**健康检查粒度**：每收到一个 token 即确认连接存活。90 秒 stale-stream 检测防止子代理和静默模式调用方无限挂起。同步路径仅在流式不可用时作为 fallback。

## 9. 后台审查模式

### 决策

记忆回顾和技能审查在**响应交付后**才启动，运行在独立线程上。

### 流程

```
模型生成响应 → 立即流式返回给用户 → 用户看到响应 → 后台线程启动审查
```

### 审查内容

| 类型 | 时机 | 影响 |
|---|---|---|
| 记忆回顾 | 每轮结束后 | 更新长期记忆索引 |
| 技能审查 | 技能使用后 | 调整技能可用性评分 |

### 技能来源追踪

技能写入操作通过 `ContextVar` 标记来源，确保审查能区分"用户直接要求"和"Agent 自主调用"。

### 为什么后台执行

用户任务优先于内部维护。审查可能耗时数秒到数十秒，阻塞主循环会严重影响体验。后台执行还提供了容错隔离：审查失败不影响主流程。

---

## 10. 精确锁定依赖（安全）

### 决策

所有直接依赖使用 `==X.Y.Z` 精确锁定，不接受任何范围（`>=`、`~=` 等）。

### 锁定机制

| 层级 | 机制 |
|---|---|
| 直接依赖 | `requirements.txt` / `pyproject.toml` 中精确版本 |
| 传递依赖 | `uv.lock` 完整锁定解析结果 |
| 可选依赖 | 提供者特定包放在 optional extras 中，按需懒加载 |

### 升级流程

```
监控安全公告 → 隔离环境测试 → 代码审查变更 → 手动更新锁定版本 → 全量回归测试
```

### 为什么精确锁定

响应供应链攻击（如 xz utils 后门事件）。浮动版本允许自动升级，可能在 CI/部署时静默引入恶意代码。精确锁定确保构建可复现，每次依赖变更都经过人工审核。
